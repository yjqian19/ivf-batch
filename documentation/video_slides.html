in IVF-based Vector Search

Reorganizing query execution, not the index

Vector search with a fixed IVF index

Vector search returns the k nearest database vectors to a query vector. IVF makes this efficient by scanning only a small set of inverted lists; our question is whether runtime scheduling can make those scans faster.

Approach

Common work changes the index structure. We keep the IVF index fixed, preserve Recall@10, and change only runtime query scheduling.

Question

When many queries arrive concurrently, can batching and reordering list scans improve throughput?

Engine: make list reuse expressible

Faiss search API

  • index.search() processes queries independently
  • Execution is query-by-query
  • Shared inverted lists are reloaded across queries

Custom search engine

  • Uses the same fixed IVF index
  • Separates centroid lookup from list scanning
  • Lets the scheduler group queries by list and reuse list loads

The goal is not to change the index; it is to expose enough control to reuse work across concurrent queries.

Validated against Faiss before scheduler experiments; Recall@10 is identical across schedulers within each workload.

Per-list batch scan: load each list once per batch

Sequential — per-query order

Each query loads its 8 lists independently. Same list re-loaded across queries.

q1: [C3, C7, C11, C42, …]
q2: [C3, C19, C11, C88, …]
q3: [C3, C7, C60, C11, …]
→ C3 loaded 3×, C11 loaded 3×

Batch — group by cluster

Pick a list, serve every query that needs it, move on.

C3: [q1, q2, q3] → L = 3 load once
C7: [q1, q3] → L = 2 load once
C11: [q1, q2, q3] → L = 3 load once
→ same arithmetic, less memory traffic

L = per-list sharing query count. For example, cluster C3 has L = 3 here.

Scan modes: MV and MM

Continue the batch example: for C3: [q1, q2, q3], the scan can run as MV or MM.

Batch(MV)

for q in [q1, q2, q3]: vecs_C3 @ q

  • Three matrix-vector scans for C3
  • Simple and low overhead
  • Works well when few queries share a list

Batch(MM)

vecs_C3 @ [q1 q2 q3].T

  • One matrix-matrix scan for C3
  • Reuses each loaded list vector across all 3 queries
  • Pays off when list sharing is high

Three schedulers on the same index

Scheduler Scan mode Batching policy
Sequential Single-query GEMV per list None
Batch(MV) Per-query GEMV per list Time + batch size
Batch(MM) GEMM per list Time + batch size

Time-window batching

Dual trigger: flush whichever fires first — Δt = 5 ms or batch size = 128. Bounds wait time at low load and memory pressure at high load.

These are the fixed parameters for the main experiment; the sweep varies both knobs later.

Experiment design: dataset, workload, device

Dataset

  • SIFT1M — 1 M vectors, 128-d, L2
  • 10,000 held-out test queries with ground truth
  • 256 clusters, nprobe = 8, k = 10

Device

  • MacBook Pro M3 Pro, 18 GB
  • Single-threaded — isolates scheduling from parallelism

Workloads

  • 10 K queries, Poisson arrivals at 2,000 QPS
  • Random: original SIFT test queries — small L
  • Clustered: 10 K arrivals concentrated around high-frequency centroids — large L

Experiment protocol: four questions, three experiments

Research questions

  • RQ1: Does batching improve throughput over sequential?
  • RQ2: How does MV vs MM depend on workload?
  • RQ3: How does batch size affect performance?
  • RQ4: What mechanism explains the MV/MM trade-off?

Experiments

  1. Main experiment (RQ1, RQ2): fixed Δt = 5 ms, MaxBS = 128, 5 runs
  2. Batch size sweep (RQ3): vary Δt and MaxBS
  3. Microbenchmark (RQ4): isolate kernels as L changes

All schedulers preserve the same Recall@10 within each workload.

Main Experiment: workload flips the winner

Figure 1. Throughput across schedulers, both workloads. Δt = 5 ms, MaxBS = 128, 5 runs.
  • Random: MV wins (+19% vs Seq); MM regresses (−3%)
  • Clustered: MM wins (+40% vs Seq, +19% vs MV)

Micro Benchmark: isolate L — crossover around L = 8

Figure 5. Synthetic list (n = 4,000, d = 128). 50 reps per point. Kernel-only, no scheduler.
  • MV is flat at 5.1–5.5 ns / (query · vector): no L dependence
  • MM improves sharply as L grows and reaches 3.77× speedup at L = 256
  • Kernel-only crossover is around L = 8, but batch-level results depend on the full L distribution
  • This explains both the main experiment and the sweep.

Takeaway and limitations

What we showed

  • Reordering query execution alone (index untouched) gives a measurable throughput improvement, with recall preserved exactly.
  • The right kernel — MV or MM — is governed by L, the per-list sharing count.
  • On our hardware, the kernel crossover is around L = 8, but this is a guide rather than an absolute boundary.

Practical guidance

Estimate the L distribution from target QPS, batch size, and query locality; use MV for low reuse and MM when list sharing is high.

Limitations & future work

  • Adaptive scan: switch MV/MM per batch from observed L — eliminates the random-workload regression
  • Multi-threading: an open question — cores already amortize memory traffic, possibly competing with our lever
  • SIFT1M only — different dimensionalities and cluster geometries remain to be tested
