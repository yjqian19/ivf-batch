# Experiment Results — Apr 14 2026

Dataset: SIFT1M (1M × 128d)  
Config: n_clusters=256, nprobe=8, k=10, arrival QPS=2000

---

## 关键结论

### 1. Recall 全部一致，正确性验证通过

三个调度器在所有参数组合下 Recall@10 均为 **0.956**，满足 ≥0.90 的目标，且结果完全一致，说明 `search_one` 和 `search_batch_per_list` 两条路径输出等价。

### 2. Sequential 基线

- **QPS: ~1994–2007**，单查询平均延迟 0.5ms，P99 0.75ms
- 延迟极低且稳定，是三者中 P99 最小的

### 3. Scheduler 2（Time-Window）：吞吐提升，延迟代价明显

- QPS 普遍高于 Sequential，最高达 **2361**（Δt=20ms, MaxBS=256）
- 代价是延迟大幅上升：avg latency 从 17ms 到 194ms，P95 最高 231ms
- **batch size 是吞吐的主要驱动力**，Δt 在 MaxBS 已饱和时影响有限：
  - MaxBS=32 时，无论 Δt 多大，AvgBS 都稳定在 ~31.9，QPS ~2100–2150
  - MaxBS=256 且 Δt≥20ms 时，AvgBS 才能充分填满（217–250），QPS 达到峰值
- 这说明在 2000 QPS 的到达率下，**时间窗口是次要瓶颈，batch size 上限才是核心约束**

### 4. Scheduler 3（Cluster-Batch）：聚合效果极差，Random workload 下反而更慢

- QPS 普遍低于 Scheduler 2，大多数配置甚至低于 Sequential（~1850–2023）
- **根本原因：AvgGrp 极小（1.1–1.7）**，即绝大多数 query 被分进了独立的 group，每个 group 单独执行一次 `search_batch_per_list`，grouping 带来的额外开销反而拖累了性能
- Groups 数量高达 6000–9000（接近 batch 中 query 的总数），说明 Random query 几乎没有 centroid 重叠，primary centroid grouping 在此 workload 下完全失效

### 5. Workload 对比：Clustered 才是 Scheduler 3 的真实场景

| Workload | Sequential | Time-Window | Cluster-Batch |
|---|---|---|---|
| Random | 1808 QPS | 2184 QPS | 1982 QPS |
| Clustered | 2007 QPS | 2329 QPS | **2346 QPS** |

- Clustered workload 下，Cluster-Batch 的 avg_group 从 1.3 跃升至 **63.7**，groups 从 7436 降至 **157**
- 此时 Cluster-Batch 与 Time-Window 几乎持平（2346 vs 2329），略有优势
- 说明 Cluster-Batch 的收益**高度依赖 query 间的 centroid 重叠度**，在现实中需要 workload 本身具有空间局部性才能发挥作用

---

## 待改进方向

1. **Jaccard grouping** 尚未在本次实验中运行（代码已实现），可能在 Random workload 下优于 primary grouping，值得补充测试
2. Scheduler 3 的 overhead 分解（quantizer time vs scan time）尚未单独计量
3. 当前 cache-locality 收益完全依赖 Python 层实现，若换成 C 扩展或 NumPy 批量化，性能上限可能更高
