# 从 Faiss 到自定义引擎：为什么调度策略毫无效果

## 1. 问题：三种调度策略，零差异

我们在 Faiss 的 `IndexIVFFlat` 之上实现了三种查询调度策略：

- **顺序执行（Sequential）**：一次处理一条查询
- **时间窗口批处理（Time-window batching）**：在时间窗口内积累查询，到期后批量提交
- **基于聚类的批处理（Cluster-based batching）**：按质心重叠程度分组后再执行

三种策略均设置 `omp_set_num_threads(1)` 以隔离调度效果。结果：**三种策略之间没有可测量的性能差异。**

---

## 2. 根本原因：Faiss 使调度策略失效

### Faiss 的查询优先、无状态架构

每次调用 `index.search(1_query, k)` 都是完全独立的操作：

```
index.search(query_A, k):
    找到 nprobe 个最近质心 → 加载并扫描对应倒排列表 → 返回 top-k → 完成

index.search(query_B, k):
    找到 nprobe 个最近质心 → 加载并扫描对应倒排列表 → 返回 top-k → 完成
```

调用之间没有任何状态传递。在处理查询 B 时，Faiss 不知道它刚刚为查询 A 扫描过第 #42 号列表，它没有跨查询上下文的概念。

### 信息丢失边界

调度层可能已经知道查询 A、B、C 都需要第 #42 号列表，并刻意将它们连续发送。但这一信息**在 `index.search()` 的 API 边界处丢失**——Faiss 的接口只接受单条查询（或内部逐条处理的批次）并返回结果。调度层精心构造的"这些查询共享列表"信号根本无法穿透这道边界。

```
调度层：  "A、B、C 共享列表 #42——把它们一起发送！"
        │
        ▼
   index.search()  ← 信息边界
        │
Faiss 内部：  独立处理每条查询，从头扫描列表
```

无论调度层如何巧妙地排列查询顺序，Faiss 都会把每条查询当作从未见过的第一条查询来处理。

### 为什么批量调用也无济于事？

调用 `index.search(batch_of_100_queries, k)` 同样没有帮助。在单线程执行下，Faiss 内部会对批次中的每条查询逐一循环，独立扫描各自的列表。批量调用本质上只是对 N 次独立单查询搜索的便捷封装——它不会在查询之间合并或共享列表访问。

---

## 3. 尝试：隔离调度变量

### 动机

第一阶段的结果引出一个问题：调度真的无关紧要，还是其效果被引擎掩盖了？为了验证这一点，我们需要一个引擎，让调度层的决策有**直接、可保证**的路径影响执行——而不是依赖可能出现也可能不出现的隐式 CPU 缓存效应。

我们的方案：构建一个自定义 IVF 引擎，将扫描顺序从查询优先改为列表优先，使调度层的分组决策直接决定消除了多少冗余列表加载。

### 实现：`CustomIVFIndex`

我们在 `engine/custom_index.py` 中构建了两种搜索方法：

**`search_one(query, k, nprobe)`** — 单查询扫描（供顺序调度器使用）：

```python
for each of the query's nprobe lists:
    load list → compute distances → collect candidates
return top-k
```

**`search_batch_per_list(queries, centroid_ids, k)`** — 按列表扫描（供批处理调度器使用）：

```python
# 构建倒排映射：list_id → [需要该列表的查询索引]
for list_id, q_indices in list_to_queries.items():
    vecs = inverted_lists[list_id]          # 只加载一次
    for q_idx in q_indices:                 # 列表数据留在缓存中
        compute distances(vecs, queries[q_idx])
return top-k for each query
```

关键区别：在 `search_batch_per_list` 中，每个倒排列表只加载一次，供所有需要它的查询共用。调度层的分组决策直接决定每次列表加载被多少查询共享。

### 自定义引擎的改变

| | Faiss `index.search()` | 自定义 `search_batch_per_list()` |
|---|---|---|
| 输入 | 单条查询（或逐条循环的批次） | 批量查询 + 质心分配 |
| 跨查询信息 | 丢失 | 保留 |
| 列表扫描方式 | 按查询，独立进行 | 按列表，跨查询共享 |
| 调度效果 | 被抵消 | 直接影响列表复用 |

### 局限性与待解问题

**该引擎并非对调度变量的纯粹隔离。** 从查询优先切换到列表优先，我们同时改变了调度策略和扫描算法。观察到的性能差异是两者共同作用的结果——无法将其单独归因于调度策略。

**该引擎并非生产设计。** 列表优先扫描不被主流向量数据库（Faiss、Milvus、ScaNN 等）采用，它以牺牲并行性和单查询延迟为代价换取批次级别的列表复用。我们在此将其作为放大调度效应的实验装置，而非对查询优先引擎的替代方案。

**三种策略现在共用同一代码路径。** 顺序、时间窗口和基于聚类的调度均经由 `quantizer_search` + `search_batch_per_list` 执行，顺序调度退化为 batch_size=1。这确保唯一的变量是调度策略本身，而非底层搜索实现。

### 一个未解决的问题：为什么查询优先模式下看不出差异？

理论上，即便在查询优先引擎中，调度顺序也应该通过隐式 CPU 缓存效应产生影响。如果查询 A 和 B 都探查列表 #42，连续处理它们应能让列表 #42 在处理 B 时仍留在 L3 缓存中。我们关于 SIFT1M 上差异未显现的假设：

- 每个倒排列表约 2MB，nprobe=8 → 单查询工作集约 16MB
- 现代 L3 缓存约 16–32MB → 无论查询顺序如何，工作集基本都能放入缓存
- 缓存复用信号真实存在，但在该数据集规模下过于微弱，无法被测量

**这一点尚未得到验证。** 我们尚未在更大数据集（如 1000万+ 向量）上运行实验（此时缓存压力会更高），也未收集硬件级缓存缺失统计数据（`perf stat`）来确认调度顺序是否影响 SIFT1M 上的缓存行为。这些是有价值的后续实验。

## 4. 调度策略与自定义引擎的映射关系

| 调度器 | 搜索方法 | 列表复用情况 |
|---|---|---|
| 顺序执行 | `search_batch_per_list(1 条查询)` | 退化为大小为 1 的批次——无跨查询共享 |
| 时间窗口 | `search_batch_per_list(batch)` | 批次成员共享列表加载 |
| 基于聚类 | `search_batch_per_list(group)` | 按质心重叠分组 → 最大化共享 |

三种策略均使用同一自定义引擎和相同代码路径（`quantizer_search` + `search_batch_per_list`），不再调用 Faiss 的 `index.search()`。唯一的变量是调度器如何组装传入 `search_batch_per_list` 的查询集合；顺序调度每次只传入一条查询。

---

## 5. 实验叙事

两阶段实验讲述了一个完整的故事：

### 第一阶段：Faiss 基线（所有策略 batch_size=1）

- 三种调度器均独立调用 `index.search(1 条查询, k)`
- 结果：无性能差异
- 结论：**Faiss 的无状态、查询优先架构使外部调度策略完全不可见**

### 第二阶段：自定义引擎

- 三种策略均通过同一代码路径使用 `search_batch_per_list()`
- 顺序调度每次传入一条查询（batch_size=1，无跨查询共享）
- 批处理策略传入更大的批次（跨查询列表共享）
- 结果：批处理策略优于顺序执行；基于聚类的分组进一步优于时间窗口
- 结论：**当引擎具备批处理感知能力时，调度策略通过列表复用直接影响性能**

### 综合结论

调度策略与引擎架构是耦合的。智能调度的收益只有在引擎被设计为能够利用跨查询结构时才能实现。这促使我们将调度器与搜索引擎作为整体协同设计，而非将两者视为独立层次。

## 6. 为什么 Faiss 不使用列表优先扫描

Faiss 的查询优先设计并非缺陷——它针对不同的目标进行了优化：

| | 查询优先（Faiss） | 列表优先（我们的引擎） |
|---|---|---|
| 并行性 | 每条查询独立 → 轻松 OpenMP 并行化 | 查询共享状态 → 需要同步 |
| 延迟 | 单条查询立即返回 | 必须等待整个批次完成 |
| 内存 | 每条查询一个 top-k 堆，完成后释放 | 批次中所有查询的累积缓冲区 |
| 适用场景 | 在线服务、多线程、GPU | 单线程、有查询重叠的批处理场景 |

我们的引擎在受控的单线程环境中展示了调度效应。在生产环境中，并行性与列表复用之间的权衡仍是一个开放的设计问题。
